Task Summary – Well Allocation Validation with 5-Month
This task focused on improving and validating the well allocation process by introducing a 5-month recompletion condition for associating production decline (ARPS) data with individual wellbores. The primary objective was to ensure that production allocation was mapped only to wells that satisfied predefined business rules based on recompletion dates, filing information, and well characteristics. The work involved analyzing large datasets stored in MongoDB, validating historical production records, refining allocation logic, and improving the overall reliability of the generated well production data.The first stage of the task involved understanding the existing allocation workflow and identifying the datasets required for processing. Multiple MongoDB collections were analyzed, including linkage information, decline analysis data, and well metadata. The relationship between these collections was studied to understand how production history, recompletion events, API numbers, and well profiles were connected. This analysis helped establish a complete workflow for validating the allocation process before implementing any modifications.The processing pipeline was designed to iterate through each well identifier and verify whether the necessary production and linkage information existed. Wells without valid decline data or linkage information were identified separately so that missing records could be tracked instead of silently failing during execution. Separate tracking lists were maintained for linkage data not found, decline data not found, and unprocessed well identifiers, allowing the entire execution process to remain transparent and auditable.A major part of the implementation involved filtering only the relevant well records required for allocation. Multiple business conditions were applied to retrieve valid records from the linkage dataset. Only wells with valid recompletion dates, supported wellbore profiles such as Horizontal, Directional, and Vertical wells, and accepted filing purposes were considered. Wells classified as shut-in producers or shut-in UIC wells were intentionally excluded to prevent inactive wells from affecting production allocation. Duplicate API records were also removed to ensure each well was evaluated only once during processing.The recompletion date validation process required additional data cleaning because historical records contained multiple date formats. Logic was implemented to detect invalid date formats, convert different representations into a standardized datetime format, and sort wells chronologically based on recompletion dates. This ensured that all subsequent date-based calculations operated consistently regardless of the original source format.Another significant portion of the task involved reconstructing production time-series data from the decline dataset. Helper functions were used to map production dates with recompletion events while maintaining chronological consistency. Each ARPS column was examined to identify its first valid production month, and these production periods were compared against the recompletion timeline of every associated well.The core enhancement implemented during this task was the introduction of the five-month validation rule. For every mapped production segment, the difference between the production start month and the well recompletion month was calculated. Production allocation was considered valid only when the production began within zero to five months after the recompletion event. If the production period fell outside this allowable window, the associated well was marked as invalid for that production segment. This validation significantly reduced incorrect production assignments that could occur because of historical overlaps or inaccurate linkage information.Additional validation logic was incorporated to identify wells that could not be confidently associated with any production segment.
