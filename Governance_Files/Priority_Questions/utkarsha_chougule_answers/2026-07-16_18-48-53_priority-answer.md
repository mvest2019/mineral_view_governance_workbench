# Priority Question Answer

Question:
Does the $226,340 estimate example use a real member's valuation, and is the MVEstimate NaN/$0 defect a prerequisite for the Estimated Value card work?

Answer:
Can't confirm — and that's exactly why it's on the open-items list. The $226,340 came through your voice memo as an example figure; I couldn't verify where it's sourced from. If the card is pulling that number from a live member record, that's also a privacy flag — the same real-data-in-a-shareable-build risk you raised on the earlier prototypes. It needs checking against the data source.
Is the MVEstimate defect a prerequisite? Yes — treat it as a hard blocker, effectively P0. The Estimated Value card spec (and especially the "Explain this estimate → show how the number is derived" drill-down, item 3.3.2) assumes a correct numeric valuation with a breakdown behind it. If valuations are stored as text and MVEstimate returns $0 for depleted leases, the card renders wrong or $0 values and the drill-down explains a broken number. So the defect should be fixed first, or run in parallel as a declared dependency — the card work shouldn't ship on top of it.
